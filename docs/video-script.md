在聊技术细节之前，
先说这个项目最重要的设计思想。

做 AI 自动化最常见的坑是什么？

让 LLM 来决定流程走向。

比如让 AI 判断这个代码合不合格，
然后根据回答决定通过还是打回。

问题是，大模型的输出是概率性的。
同一段代码今天可能说通过，
明天可能说有安全隐患。

MasterCoder 的解法很干净：
把流程控制和内容生成彻底分离。

流程靠代码里写死的状态机，
质量靠客观的门禁工具。

**AI 负责思考和创造，
代码负责判断和执行。**

>>> REQ 状态机流转图
**[动画]**
深色背景，标题「需求状态机」。
画面中央横向排列六个圆角矩形节点，用箭头连接：
PENDING → READY → DEVELOPING → REVIEWING → TESTING → DONE
每个节点用不同颜色：灰、蓝、黄、紫、橙、绿。
DEVELOPING 下方有一条虚线分支指向「FIXING」节点，
FIXING 有回指箭头指回 DEVELOPING，旁边标注「重试 ≤ 3 次」。
FIXING 还有一条红色箭头指向「BLOCKED」节点。
动画：节点从左到右依次亮起，每个间隔 0.8 秒；
分支箭头在主链亮完后出现。

接下来看核心数据结构：状态机。

每个需求会经历这些状态。

从 **PENDING** 等待依赖，
到 **READY** 可以开始，
然后 **DEVELOPING** 写代码，
**REVIEWING** 审查代码，
**TESTING** QA 验收，
最后 **DONE** 合并完成。

任何环节失败，进入 FIXING 重试。
超过三次，标记为 BLOCKED。

>>> 状态文件 JSON 示例
**[代码]**
深色编辑器风格，左侧有行号。
显示以下 JSON 内容，逐行出现：
```json
{
  "req_id": "REQ-03",
  "title": "实现 API 客户端",
  "blocked_by": ["REQ-02"],
  "state": "READY",
  "retries": 0,
  "max_retries": 3,
  "branch": "feat/req-03-api-client",
  "pr_number": null,
  "last_error": null
}
```

所有状态保存在一个 JSON 文件里。

没有用数据库，没有用 Redis，
就是一个 JSON 文件。

好处是：任何人都能打开看状态，
改一个字段就能干预流程，
进程挂了重启后继续跑就行。

`blocked_by` 字段实现了依赖管理。

REQ-03 依赖 REQ-02，
那 REQ-02 没完成之前，
REQ-03 就不会被启动。

>>> Orchestrator 推进逻辑
**[动画]**
深色背景，标题「编排引擎 Orchestrator」。
画面中央是一个大的流程图，纵向排列：
顶部：「_advance(req)」函数入口
然后四个判断分支，每个是菱形判断框 + 矩形操作框：
1. state == READY? → 调用 Dev Agent
2. state == DEVELOPING? → 校验门禁 → 推进到 REVIEWING
3. state == REVIEWING? → 调用 Review Agent
4. state == TESTING? → 调用 QA Agent → gh pr merge → DONE
右侧有一个红色分支：「失败 → retries++ → BLOCKED?」
动画：从顶部开始，每个分支依次高亮（0-6s）。

**Orchestrator** 是整个系统的大脑，
大约四百三十行代码。

它的逻辑其实很简单：
根据当前状态，决定下一步做什么。

状态是 READY？调用 Dev Agent。
开发完成？跑门禁，推进到审查。
审查通过？调用 QA Agent。
全部通过？自动合并 PR。

每个需求有独立的重试计数，
一个需求卡住不会拖垮整个队列。

第五步，通过后 push 并创建 PR。

Dev 有推送权限，
Review 有审批权限，互不交叉。

- 图标：机器人
右栏标题「严格真人模式」（绿色），纵向流程：
- 系统轮询 GitHub API（每 30 秒）

审查环节支持两种模式。

输出格式是严格的 JSON。

编排器解析 JSON 里的 verdict 字段，
决定通过还是打回。
不是去理解自然语言回复。

严格真人模式下，
系统每三十秒轮询一次 GitHub。
等待真人点 Approve 或者评论。

两种模式可以自由组合。

Review 用 AI 自动审，
QA 用真人验收，完全可以。

你对 AI 有多信任，
就给它多少自主权。

>>> 需求文档结构示例

2.「功能规格」→ 编号列表：1. 命令名为 mastercoder  2. 打印欢迎信息...

需求文档的写法本身就值得学习。

功能规格精确到行为细节，
比如：打印欢迎信息，
显示输入提示符大于号。

这种精度的文档不是为人写的。
但对 AI Agent 来说，
模糊的需求是灾难性的。

精确的规格才是它正确工作的前提。

>>> 经验总结第一条
**[文字卡]**

最后总结几个核心经验。

第一，AI 系统的可靠性，
不来自更强的模型，
而来自更好的约束。

MasterCoder 默认用的是智谱 GLM。
但通过门禁和状态机，
把 AI 的不确定性控制在安全范围内。

>>> 经验总结第二条
**[文字卡]**
深色背景，顶部小字「经验 2」，居中显示：
> 「简单方案往往是最好的方案」

第二，简单方案往往是最好的方案。

JSON 文件做状态存储。
可读、可编辑、可版本控制、
不需要额外基础设施。

如果一开始就上数据库，
反而增加了部署复杂度。

>>> 经验总结第三条
**[文字卡]**
深色背景，顶部小字「经验 3」，居中显示：
> 「人机协作需要连续的信任光谱」
> 「而不是非此即彼的开关」
白色文字，淡入停留 3 秒。

第三，人机协作需要连续的信任光谱。

刚开始用可以全部真人审批。
跑一段时间建立信任后，
再逐步切换到 AI 自动审查。

这比全自动或全手动要务实得多。
